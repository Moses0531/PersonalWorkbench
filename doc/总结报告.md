个人总结报告
一、遇到的最大困难及解决过程
1.单人多窗口下的 Git 冲突：并行开发的代价与应对
（1）问题描述：
迭代节奏快时，需同时修改多个页面UI。因采用单人多窗口并行开发（多个编辑器窗口同时编辑不同功能模块），常在同一批 Vue 组件、路由或权限菜单中产生 Git 冲突。例如，同一组件中既需调整布局，又要变更接口字段，还夹杂样式变量修改。最棘手的问题在于：冲突本质是同一文件被多窗口“并行修改”，且背后往往存在两套合理的修改意图——例如，一边需合并任务与项目入口以优化体验，另一边需坚持目录结构约定，避免因临时拆分（如错误创建目录）导致后续维护成本升高，也会产生git冲突，页面的效果往往也达不到预期要求。
（2）解决策略：
先理清任务的核心目的，点名修改的文件范围，明确告诉AI修改哪几个页面，在该范围的页面等待任务结束以后再对齐进行修改。
（3）经验总结
单人开发中的 Git 冲突本质是并行开发与版本控制不匹配。需要理清任务目标。后续如果是多人协作场景，更有注重团队的沟通，避免出现与团队成员做了相同工作，甚至是重新构建模块逻辑，影响开发效率。
2.UI 设计：从堆功能到讲理清页面
（1）问题描述
前期容易界面陷入"接口有了就往页面塞"的误区：列表、筛选、状态、附件、AI 入口同时出现，密度高但看不出产品主张，不同模块之间的不同内容视觉对比度不明显。
（2）解决策略
明确这一个页面要让用户做什么？哪些信息是主要信息，哪些页面是次要信息。例如在登录页设计上通过办公电脑等象征性元素告诉用户这是一个工作台系统。
（3）经验总结
要认真的分析页面功能，可以多参考一些主流项目的页面是如何设计功能的，取长补短。
3.UI样式
（1）问题描述
ai生成的界面在大多数情况下设计较为呆板，也缺乏统一性，例如用户管理界面采用红色系，而且在个人中心页面采用黄色系，导致了用户视觉效果的不一致；很大情况下ai设计页面样式难以达到预期效果。
（2）解决策略
安装一些常见的skill约束ai，让ai对应这些skill来执行任务；也可以先用生成图片工具给出一份参考图让ai执行；最后就是描述的指令要精确；例如在首页添加时钟样式，采用与首页相同的配色方案。
（3）经验总结
在让ai写页面样式时一定要用skill，也不要局限于单一的模型，不同模型对不同内容的理解存在偏差，例如Gemini的审美比大部分ai要好，在设计样式时可以优先选择。
二、对 AI 辅助编程的心得体会
1.AI 很擅长在现有技术栈上铺页面、补 API、改样式和写文档，但不会替开发者做产品判断。用得好的前提是：自己先有构思，再把问题说清楚——范围、约束、参考页、不要什么，比“帮我优化一下”有效得多。
2.描述问题比催促输出更重要：报错原文、复现路径、期望行为写全，才对准根因；含糊需求只会换来含糊的 UI。
3.AI 是加速加速工具，不是架构师：目录结构、权限模型、是否合并页面、安全边界必须人拍板；方案要带着批判验收，使用的技术也是要人来定夺，不能完全依赖ai。
三、个人具体贡献
（1）项目核心功能与架构设计。确定了项目需要哪些功能，选用什么技术模型来实现这个项目，以及这个项目面对的对象群体。
（2）信息架构与交互设计。主导任务与项目入口合并方案评估，推动“导航整合但数据模型分离”的架构决策；仔细打磨了登录页与首页的样式设计。
（3）视觉与品牌一致性：参与首页、登录页、悬浮 Dock、图标与配色统一工作，减少模板感，提升产品质感。
（4）测试项目功能，模拟用户的日常使用，亲自锁定交互性较低的地方，并将其进行优化，例如增加Tab键的检索，将相关文件存入该系统，以此增强用户体验。
